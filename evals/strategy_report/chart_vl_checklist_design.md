# Chart VLM Checklist Judge Design

This document specifies the next chart-evaluation iteration for strategy-report QA. The goal is to make the VLM judge auditable, cheaper to calibrate, and better aligned with human financial-report reviewers.

## 1. Scope Split: What VLM Should And Should Not Judge

The chart QA module should not ask the VLM to solve every chart problem. Each evaluation item should be routed to the cheapest reliable method.

| Evaluation item | Primary method | VLM role | Notes |
| --- | --- | --- | --- |
| Chart existence and screenshot availability | deterministic extractor | none | PDF/HTML extractor records image paths and candidate counts. |
| Crop completeness | image heuristics + VLM | confirm/diagnose | Heuristics catch blank/tiny/aspect issues; VLM says whether the target visual is complete. |
| Title/unit/source/date presence | parser/rules first | visual fallback | Use extracted text first; VLM checks visible labels when OCR/text extraction misses them. |
| Data faithfulness to source data | structured numeric check | limited | If underlying data table exists, do not ask VLM to verify exact numbers. VLM only flags visibly impossible or contradictory cases. |
| Key numeric readability | VLM | primary | VLM can judge whether labels, axes, and legends are readable enough for a human reviewer. |
| Chart-text alignment | VLM + text retrieval | primary | Use target visual + full-page text; VLM identifies matching spans and checks support/contradiction. |
| Chart type appropriateness | VLM | primary | VLM judges whether the visual form suits the analytical task. |
| Financial professionalism | VLM | primary | Includes restraint, labeling, scale choices, clutter, and misleading design. |
| Layout overflow/overlap | DOM/screenshot rules + VLM | confirm/diagnose | Rules should detect obvious overflow; VLM explains visual impact. |
| Decorative/non-analytical visual | rules + VLM | confirm | Rules filter logos/headshots; VLM catches remaining decorative charts. |

Important boundary: `data_faithfulness_score` should become a composite score, not a pure VLM score. It should combine structured numeric checks when available, rule-based title/unit/source checks, and VLM contradiction flags. The VLM should not be rewarded or punished for failing to read exact small numbers that a parser can extract more reliably.

## 2. Checklist Strategy

The VLM receives two kinds of checklist items:

1. **Universal checklist**: always applied to financial visualizations.
2. **Contextual checklist**: generated by the VLM after inspecting the chart type, page context, and report purpose.

The judge must score each checklist item and cite evidence. This makes the score reviewable in the dashboard and suitable for human-alignment experiments.

### Universal Checklist

Each item is scored from `0` to `1`, with `not_applicable` allowed only when the item truly does not apply to the visual type.

| ID | Item | What To Check | Preferred Method |
| --- | --- | --- | --- |
| U1 | Analytical purpose | The visual answers a clear research question and is not decoration. | VLM |
| U2 | Crop/render completeness | Target screenshot contains the full intended chart/table, not only a legend, footnote, sidebar, or cropped fragment. | heuristic + VLM |
| U3 | Title and analytical framing | The title/caption states what the chart is about, not just a generic label. | rules + VLM |
| U4 | Unit, scale, and time window | Units, percentages/bps/currency, axis scale, period, and date range are visible or stated nearby. | rules + VLM |
| U5 | Source and methodology note | Data source and important footnotes are visible or available in surrounding text. | rules + VLM |
| U6 | Readability | Axis labels, legends, series names, table headers, and key labels are readable at dashboard resolution. | VLM |
| U7 | Chart type suitability | Chart form matches the analytical task: trend, comparison, distribution, allocation, scenario, correlation, or table lookup. | VLM |
| U8 | Visual professionalism | Styling is restrained, non-cluttered, non-misleading, and suitable for institutional research. | VLM |
| U9 | Text binding | Full-page text includes at least one relevant explanation of the current visual. | VLM |
| U10 | Claim support | The matched text is supported by the visual and does not contradict it. | VLM |
| U11 | Risk of visual misdirection | Scale, axis truncation, color emphasis, ordering, or missing baseline does not mislead the reader. | VLM |
| U12 | Decision usefulness | The visual contributes to a strategic investment/research conclusion rather than restating obvious facts. | VLM |

### Contextual Checklist Generation

After applying universal checks, the VLM adds `2-5` contextual checklist items. Examples:

- For time-series charts: start/end period, trend direction, inflection points, volatility, and recency are clear.
- For bar/ranking charts: category ordering, top/bottom entities, and comparison baseline are clear.
- For stacked/allocation charts: components sum logically and legend/category mapping is clear.
- For tables: headers, row/column labels, units, and highlighted conclusions are readable.
- For scenario/fan charts: base/upside/downside assumptions and probability language are clear.
- For valuation/market charts: multiples, yield, spread, index, and benchmark definitions are explicit.
- For multi-panel pages: the current visual is distinguished from neighboring visuals and matched to the correct text.

Contextual items must be concrete, not generic. Bad: "chart is good". Good: "For this time-series chart, the start and end dates are legible enough to support a 2020-2026 trend claim."

## 3. VLM Output Schema

The VLM must return strict JSON with no markdown.

```json
{
  "ok": true,
  "chart_id": "string",
  "chart_type": "line|bar|stacked_bar|area|scatter|table|matrix|multi_panel|mixed|unknown",
  "is_analytical_visual": true,
  "target_visual_description": "short description of what is visible in the target screenshot",
  "main_takeaway_from_visual": "one sentence",
  "visible_metadata": {
    "title": {"value": "string", "confidence": 0.0},
    "unit": {"value": "string", "confidence": 0.0},
    "time_window": {"value": "string", "confidence": 0.0},
    "source": {"value": "string", "confidence": 0.0}
  },
  "matched_text_spans": [
    {
      "span_id": "t001",
      "text": "short quote or paraphrase from provided page text",
      "relevance": 0.0,
      "relationship": "explains|mentions|supports|contradicts|unrelated",
      "reason": "why this span is the best match for the current visual"
    }
  ],
  "universal_checklist": [
    {
      "id": "U1",
      "label": "Analytical purpose",
      "score": 0.0,
      "status": "pass|partial|fail|not_applicable",
      "evidence": "specific visible or textual evidence",
      "severity_if_failed": "none|low|medium|high"
    }
  ],
  "contextual_checklist": [
    {
      "id": "C1",
      "label": "context-specific criterion",
      "score": 0.0,
      "status": "pass|partial|fail|not_applicable",
      "evidence": "specific evidence",
      "severity_if_failed": "none|low|medium|high"
    }
  ],
  "subscores": {
    "crop_completeness": 0.0,
    "metadata_completeness_visual": 0.0,
    "readability": 0.0,
    "chart_type_suitability": 0.0,
    "chart_text_alignment": 0.0,
    "claim_support": 0.0,
    "visual_professionalism": 0.0,
    "decision_usefulness": 0.0
  },
  "hard_flags": [
    {
      "flag_type": "decorative_visual|incomplete_crop|unreadable|missing_critical_unit|missing_critical_source|text_contradiction|misleading_scale|wrong_chart_type",
      "severity": "low|medium|high",
      "evidence": "why this flag applies"
    }
  ],
  "overall_vlm_visual_score": 0.0,
  "confidence": 0.0,
  "review_notes": "short reviewer-facing explanation"
}
```

Schema notes:

- `overall_vlm_visual_score` is not the final chart score. It is the VLM visual-communication score.
- `chart_text_alignment` and `claim_support` should be based on `matched_text_spans`, not on assumed nearby text.
- Exact numeric correctness should only be judged when numbers are plainly visible. Otherwise the VLM should write an uncertainty note instead of guessing.
- Any high-severity `text_contradiction`, `incomplete_crop`, or `decorative_visual` flag should trigger a hard penalty in the downstream aggregator.

## 4. Prompt Draft

System prompt:

```text
You are a senior financial research chart reviewer. Your job is to evaluate whether a chart/table in a strategy report is useful, readable, professionally presented, and aligned with the report text.

You are not a general report grader. Focus only on the target visualization. Use the full-page screenshot and page text only to understand context and find the text that belongs to the current visual.

Do not invent exact values when labels are too small. If exact numeric verification requires source data or OCR beyond what is visible, state the limitation and judge readability/support instead.

Return strict JSON only. Scores are continuous from 0 to 1.
```

User prompt template:

```text
Evaluate TARGET_VISUAL_SCREENSHOT as the current visualization.

Inputs:
- TARGET_VISUAL_SCREENSHOT: the crop/object under review.
- FULL_PAGE_SCREENSHOT: the complete page when available. Use it for layout and context.
- PAGE_TEXT_BLOCKS: extracted page text spans. On multi-chart pages, find the span(s) that best match this target visual.
- RULE_HINTS: parser/rule-detected title, unit, source, numbers, dates, crop quality, and expected chart match.

Evaluation boundaries:
1. Apply the universal checklist U1-U12.
2. Add 2-5 contextual checklist items specific to this visual type and page situation.
3. Give evidence for every failed or partial item.
4. For chart-text alignment, first identify matched_text_spans from PAGE_TEXT_BLOCKS, then judge whether those spans explain/support/contradict the visual.
5. Do not perform exact data-faithfulness grading unless the relevant values are plainly visible. Exact numeric faithfulness is handled by structured checks outside the VLM.
6. Penalize incomplete crops, unreadable labels, missing critical units/source, decorative visuals, misleading chart forms, and contradictions with matched text.
7. If the full page contains multiple charts, do not assume the nearest text belongs to the target visual.

Universal checklist:
U1 Analytical purpose
U2 Crop/render completeness
U3 Title and analytical framing
U4 Unit, scale, and time window
U5 Source and methodology note
U6 Readability
U7 Chart type suitability
U8 Visual professionalism
U9 Text binding
U10 Claim support
U11 Risk of visual misdirection
U12 Decision usefulness

Return JSON matching the provided schema.

SCHEMA:
{schema_json}

CONTEXT:
{context_json}
```

## 5. Score Aggregation Proposal

The chart module should expose three layers:

1. **Rule/numeric layer**
   - inventory
   - screenshot exists
   - crop geometry
   - title/unit/source/date text hints
   - structured numeric consistency when available

2. **VLM checklist layer**
   - crop completeness confirmation
   - readability
   - chart type suitability
   - visual professionalism
   - text binding and claim support
   - decision usefulness

3. **Final chart score layer**
   - combines rule/numeric and VLM outputs
   - applies hard penalties
   - emits issue taxonomy for skill iteration

Suggested final per-chart weights:

| Component | Weight |
| --- | ---: |
| Metadata completeness from rules + VLM fallback | 0.15 |
| Structured numeric/data faithfulness | 0.20 |
| Crop/render completeness | 0.12 |
| Readability and visual professionalism | 0.18 |
| Chart type suitability | 0.10 |
| Chart-text alignment | 0.15 |
| Decision usefulness | 0.10 |

Hard penalties:

- `text_contradiction: high`: final per-chart score capped at `0.45`.
- `decorative_visual: high`: capped at `0.40`.
- `incomplete_crop: high`: capped at `0.50`.
- `unreadable: high`: capped at `0.55`.
- Missing source/unit on a key chart: subtract `0.08-0.15`, depending on severity.

Report-level chart score should not be a plain mean. Use a weighted mean with higher weight for charts that:

- match expected golden metadata,
- are referenced by page/report text,
- contain many key facts/numbers,
- are near executive-summary or thesis sections,
- are the only chart supporting a major claim.

## 6. Human Alignment Set

After this prompt is implemented, export `30-50` chart review packets. Each packet should contain:

- `chart_id`, `case_id`, source format, page/object index.
- target visual screenshot.
- full-page screenshot.
- extracted page text blocks.
- rule hints: title, unit, source, numbers, dates, crop quality.
- universal checklist U1-U12.
- blank contextual checklist slots for human additions.
- VLM checklist output, when available.

Human annotators should score:

- crop completeness: `0-2`
- readability: `0-2`
- metadata completeness: `0-2`
- chart type suitability: `0-2`
- chart-text alignment: `0-2`
- claim support: `0-2`
- visual professionalism: `0-2`
- decision usefulness: `0-2`
- hard flags: yes/no with severity
- overall grade: `A/B/C/D`

For alignment analysis:

- Convert human `0-2` items to `0/0.5/1`.
- Compute VLM-human absolute error per checklist item.
- Compute adjacent grade agreement for overall grade.
- Track high-severity issue recall and false-positive rate.
- Run each VLM packet twice on a 10-sample subset to estimate stability.

Initial acceptance targets:

- High-severity visual issue recall >= `85%`.
- Good-chart false-positive rate <= `15%`.
- Adjacent overall-grade agreement >= `80%`.
- Mean absolute checklist error <= `0.18`.
- Repeat-run standard deviation of `overall_vlm_visual_score` <= `0.08`.

## 7. Implementation Steps

1. Add a new checklist prompt/schema path in `chart_judges.py`.
2. Keep the old judge output parser temporarily for backward compatibility.
3. Update `chart_qa.py` to consume `subscores`, `universal_checklist`, `contextual_checklist`, and `hard_flags`.
4. Update the dashboard to show checklist rows with evidence and matched text spans.
5. Add an export script for human packets:
   - input: eval result directory
   - output: static HTML/JSONL review packets
   - sample size: configurable, default `40`
6. Run a low-cost smoke test:
   - 2 PDF charts
   - 2 HTML charts
   - 1 multi-chart page
7. After prompt stability, export 30-50 packets for human annotation.

## 8. Open Calibration Questions

- Should missing source/unit be a hard issue for all charts, or only for key charts?
- Should original benchmark reports be graded more leniently than generated reports for website-style charts?
- How should tables be weighted relative to charts when a table carries the main evidence?
- How many contextual checklist items give the best signal without making annotation too slow?
